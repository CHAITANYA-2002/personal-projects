Step 2 — Diagnose the Mechanism (per column)
Analysis · Run all 3 tests for each column with missing values

Why Is This Column Missing? — 3 Diagnostic Tests

Understanding why data is missing is the most critical step. The same imputation applied to different mechanisms produces different results — some misleading.

Test 1 — Pattern Analysis

Missingness Map

Create a binary indicator: 1 = missing, 0 = present. Visualise it across the dataset rows.

🔎 Random scatter → MCAR signal
🔎 Clusters in rows/time → MAR or MNAR
Test 2 — Feature Dependency

Compare Groups

Split rows into "missing" vs "not missing" groups. Compare distributions of other features across both groups.

🔎 <5% difference → weak signal (MCAR likely)
🔎 10–30% difference → strong MAR signal
Test 3 — Target Dependency

Check the Target Variable

Compare the target (y) distribution for missing vs. non-missing rows in this column.

🔎 <5% target diff → no strong signal
🔎 >10% target diff + domain logic → MNAR
MCAR

Missing Completely At Random

The probability of a value being missing is independent of all variables — observed or unobserved. Missingness is a random accident.

✓ All 3 tests show no pattern
✓ Missing rows look like non-missing rows
✓ No target difference detected
Examples: Sensor dropout, survey respondent skipped randomly, manual data entry error
MAR

Missing At Random

Missingness depends on other observed columns, but not on the missing value itself. Systematic, but explainable using available data.

⚠ Missing rate differs across groups
⚠ Explainable by another feature
⚠ Some target difference via groups
Examples: Income missing more for self-employed; test results skipped for outpatients
MNAR

Missing Not At Random

Missingness depends on the missing value itself. People or systems suppress the value because of what it reveals.

🚨 Strong target association
🚨 Domain reasoning confirms suppression
🚨 Cannot be explained by other columns
Examples: Low income not reported; severe illness skipping a test; credit-risky customer hiding data

Step 4 — Choose a Handling Strategy
Decision · Match strategy to mechanism + missing %

Strategy Selection Rules

There is no single correct imputation method. The right choice depends on the missingness mechanism, data type, missing %, and available features. Use the strategy cards below.

🗑 Drop Rows / Column MCAR only

Remove rows where the value is missing. Only safe when missingness is MCAR and loss is small.

Drop Rows

Remove entire row. Safe only if missing ≤ 5% and MCAR is confirmed.

✓ No artificial data introduced

✗ Loses information if overused

Drop Column

Remove entire column if missing > 40–50% and the feature has low predictive value.

✓ Eliminates noise

✗ Irreversible — verify with domain expert first

🚩 Missing Indicator Flag MAR ≥10% · All MNAR

Add a new binary column (1 = was missing, 0 = was present) before imputation. This preserves the information that the value was missing.

When to Always Add

• Any MNAR column (mandatory)
• MAR columns with missing ≥ 10%
• Any column where missingness correlates with target

✓ Lets model learn from the fact of missingness
✓ Prevents signal loss when imputing

📊 Statistical Imputation — Simple Methods MCAR · MAR (low %)

Replace missing values using a single summary statistic computed from non-missing rows. Train-set statistics only — never compute on full dataset before splitting.

Mean Imputation

Replace with column mean. Best for symmetric, normally distributed numerical data with no outliers.

Use when: MCAR, symmetric dist.

✗ Reduces variance; sensitive to outliers

Median Imputation

Replace with column median. Robust to outliers and skewed distributions. Preferred over mean in most cases.

Use when: MCAR/MAR, skewed data

✓ Outlier-resistant; recommended default

Mode Imputation

Replace with most frequent value. Used for categorical or ordinal columns.

Use when: Categorical columns, MCAR

✗ Can over-represent dominant category

Arbitrary / Constant

Replace with a fixed value like 0, -1, -999, "Unknown", or "Missing". Must have semantic meaning.

Use when: MNAR or when flag is needed

✗ Can mislead model if value is meaningless

🤖 Advanced Imputation — Model-Based Methods MAR (moderate–high %)

Use relationships between features to predict missing values. These methods respect feature interactions but carry risks of data leakage and computational cost. Never include the target variable as a predictor.

KNN Imputation

Find the K most similar rows (by other features) and impute using their values. Captures local structure in data.

Use when: Well-scaled features, strong local similarity, moderate missing %

✓ Preserves local patterns

✗ Slow on large datasets; sensitive to scale and noise; suffers in high dimensions

Key rule: Always scale features before applying KNN imputer

Iterative Imputer
(MICE)

Models each column with missing values as a function of all other columns, iterating until convergence. Uses any regression model internally.

Use when: Multiple columns are missing, strong inter-feature correlations, MAR confirmed

✓ Accounts for relationships between all features
✓ Most statistically principled approach

✗ Expensive to compute; risk of instability; requires careful convergence check

Group-wise Imputation

Compute mean/median within subgroups defined by a related categorical feature. Respects group-level distributions.

Use when: MAR — missingness depends on a known group variable

✓ Preserves group-level variance
✓ Simple and interpretable

✗ Group sizes must be large enough to compute stable statistics

Example: Impute salary by employment type; impute age by education band

🚩

Reminder — Missing Indicator Must Be Created BEFORE Imputation

Once you impute, the information that a value was missing is permanently erased. Always create the binary indicator column first, then impute. For MNAR columns the indicator is mandatory — it may become one of your most important features.

MNAR → indicator always required MAR ≥ 10% → indicator strongly recommended MCAR < 5% → indicator optional Create flag BEFORE imputing, not after Fit imputer on train set only
Step 5 — Validate Your Strategy
Checks · Run after every imputation decision — not optional

6 Validation Checks Before Accepting Any Strategy

A strategy that looks correct can still distort the data silently. These checks confirm that imputation preserved the data's statistical properties and did not introduce leakage or bias.

1
Distribution Check

Plot histograms before and after imputation. The shape should not change significantly.

Mean shift ≤ 5%
Median shift ≤ 3%
No new extreme values
2
Variance Check

Imputation artificially reduces variance because all missing values get the same or similar value.

≤ 10% change → Excellent
10–20% → Acceptable
≥ 30% → Over-smoothing 🚨
3
Correlation Check

Imputation should not change how features relate to each other or to the target.

Δ correlation ≤ 0.05
Sign must not flip
Check feature–target corr too
4
Data Leakage Check

Imputation statistics (mean, median, KNN neighbours) must be computed on training data only, then applied to test data.

Split BEFORE imputing
Never fit on full dataset
Use Pipeline in sklearn
5
Model Cross-Validation

Compare model performance with different strategies. Check stability, not just average accuracy.

Performance Δ ≤ ±1–2%
Low variance across folds
Consistency preferred over peak
6
Missing Indicator Check

If you added a missing indicator, check whether the model assigns it importance. High importance confirms MNAR.

Importance > 0 → missingness is informative
High importance → MNAR confirmed
Zero importance → consider dropping flag
Common Pitfalls — Avoid These Mistakes
Warnings · Mistakes that invalidate even the best model

⚠ 6 Most Common Pitfalls

✕

Treating all columns identically. Each column has its own mechanism. One-size-fits-all imputation is wrong by default.

✕

Imputing before train-test split. This leaks test-set statistics into training, producing inflated performance metrics.

✕

Using the target variable to impute. Including y in your imputation model causes data leakage and overfit.

✕

Assuming MAR when MNAR is likely. Overconfident imputation of MNAR values creates false signals and biased predictions.

✕

Ignoring the missing indicator. For MNAR, the fact of missingness is the signal. Imputing without a flag destroys it permanently.

✕

Validating only with accuracy. Accuracy can stay the same while variance collapses. Always check distribution, variance, and correlation too.

↻

Repeat Steps 2–5 for Every Column Independently

Each column in your dataset is a separate decision. One column may be MCAR (drop rows), another MAR (KNN imputation), another MNAR (indicator + conservative median). Work column by column — never apply one method to all columns at once.
